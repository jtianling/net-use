## Context

当前实现已支持“按监控目标缓存地址历史”，但缓存仅存在于进程内存中。用户退出 `net-use` 后再次进入，之前采集的目标地址历史无法恢复。该变更需要把该缓存提升为工作目录文件持久化，同时保持现有按 `MonitorTarget` 隔离的行为。

约束：
- 不引入远程存储或系统级数据库；
- 不改变 TUI 操作路径（选择应用 -> 监控 -> 返回/退出）；
- 文件读取失败或内容损坏时，程序不能崩溃。

## Goals / Non-Goals

**Goals:**
- 在启动 TUI 时，从运行命令的当前工作目录读取历史文件并恢复各目标缓存。
- 在监控视图结束（返回选择页或退出）后，把最新缓存写回工作目录文件。
- 继续保证不同目标之间缓存隔离，不发生地址串用。
- 对文件缺失、读写异常、JSON 损坏采用降级策略，不影响监控主流程。

**Non-Goals:**
- 不新增历史管理交互（清理按钮、历史浏览页、TTL 配置）。
- 不改变目标身份规则（PID/Name/Bundle 的命中语义保持不变）。
- 不实现跨目录共享历史（仅限“当前工作目录”）。

## Decisions

### 1) 使用工作目录固定文件名存储历史

选择：使用 `current_dir()/.net-use-address-history.json` 作为持久化位置。

备选：
- 放在 `$HOME` 全局目录；
- 放在系统临时目录。

理由：需求明确指定“运行命令的工作目录”；按目录隔离可避免不同项目/场景互相污染。

### 2) 存储结构采用可序列化的条目列表

选择：引入 `PreservedStore { entries: Vec<PreservedStoreEntry> }`，其中每个条目包含 `target + data`。

备选：
- 直接序列化 `HashMap<MonitorTarget, PreservedData>`；
- 自定义字符串键（如 `pid:123`）。

理由：条目结构更稳定、易调试；避免对 `HashMap` 序列化细节的依赖和键编码耦合。

### 3) 写入策略为“快照写临时文件后原子替换”

选择：`write(temp) -> rename(temp, target)`。

备选：
- 直接覆盖目标文件；
- 事件级增量写入。

理由：覆盖写入在中断时更容易留下半写文件；临时文件替换可降低损坏概率，且实现简单。

### 4) 异常处理采用“记录错误并回退为空缓存”

选择：读取失败、解析失败时返回空缓存并继续运行；写入失败时仅告警，不中断 TUI。

备选：
- 将错误上抛并终止；
- 强制提示用户交互确认。

理由：地址历史是增强能力，不应阻断核心监控流程。

## Risks / Trade-offs

- [历史文件持续增大] -> 监控目标/地址很多时文件可能变大。Mitigation: 当前只存去重后的字符串；后续可增加上限或压缩策略。
- [工作目录差异导致“找不到历史”] -> 用户在不同目录启动会得到不同历史集合。Mitigation: 在规范中明确“按工作目录隔离”行为。
- [写入权限不足] -> 某些目录不可写。Mitigation: 保持降级，输出错误信息并继续监控。

## Migration Plan

1. 在 `MonitorTarget` 上补齐序列化派生，保证可落盘。
2. 在 TUI 主循环初始化阶段加载工作目录历史文件，填充内存缓存。
3. 在每次监控视图结束后更新缓存并写回历史文件。
4. 增加持久化 round-trip 单测与异常场景测试（至少覆盖缺失文件/可读文件）。

回滚：如出现异常，可移除文件持久化路径并回退为会话内缓存，不影响监控主流程。
