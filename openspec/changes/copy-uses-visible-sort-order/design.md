## Context

当前监控页的地址渲染已经支持两种顺序模式（发现顺序 / 排序后顺序），并且 `O` 会即时切换屏幕显示；但 `C` 复制路径仍使用固定默认顺序构建输出文本。结果是“看见的列表顺序”和“复制到剪贴板的顺序”不一致，特别是在用户切换到排序后顺序进行人工核对时会造成误判。

该改动集中在 TUI 监控页：键盘事件处理、用于渲染的地址顺序计算复用，以及复制导出拼装逻辑。

## Goals / Non-Goals

**Goals:**
- `C` 复制结果与当前屏幕地址顺序保持一致。
- 复制行为与当前显示模式一致（masked/raw），并保持既有输出格式（每行一个条目）。
- 在现有排序切换逻辑上复用顺序来源，避免渲染与复制维护两套规则。

**Non-Goals:**
- 不改变 `O` 切换行为本身。
- 不改变 `E` 导出文件语义（除非其实现当前与复制共用同一顺序构建函数并自然受益）。
- 不改变地址采集、去重或持久化策略。

## Decisions

1. 复制内容改为基于“当前视图顺序”构建。
- Decision: 在监控页 copy handler 中复用当前 order mode 下的地址列表（与渲染同源）生成剪贴板文本。
- Rationale: 单一顺序来源可消除视图与复制不一致问题。
- Alternative considered: 复制后再二次按 mode 重排原始集合。Rejected，因为容易与渲染路径漂移，未来改动风险更高。

2. 顺序计算抽成可复用 helper（若现有 helper 已存在则直接复用）。
- Decision: 将“按当前模式得到 IPv4/IPv6 有序切片”作为独立函数供渲染与复制共享。
- Rationale: 降低重复逻辑，测试可直接验证同一函数输出。
- Alternative considered: 在 copy handler 内内联复制一份排序分支。Rejected，因为会再次引入一致性风险。

3. 为拷贝顺序增加回归测试。
- Decision: 新增/更新测试覆盖“切换为排序模式后复制”的输出顺序断言。
- Rationale: 该问题由行为回归导致，测试应直接锁定用户可见结果。
- Alternative considered: 仅做手工验证。Rejected，因为后续修改排序逻辑时容易再次回归。

## Risks / Trade-offs

- [渲染与复制复用后，helper 改动会同时影响两个路径] -> Mitigation: 增加针对渲染顺序与复制顺序的一致性测试。
- [排序策略未来扩展（例如 locale/collation）可能影响历史预期] -> Mitigation: 将期望显式写入测试，变更时同步更新 spec 与测试。
- [显示模式与复制输出强绑定后，未来若新增显示模式需要同步覆盖复制路径] -> Mitigation: 通过 `current_*_entries` 统一入口并增加 raw/masked 的组合测试。

## Migration Plan

1. 定位监控页 `C` 键处理及剪贴板文本拼装逻辑。
2. 让 copy handler 使用当前 order mode 的有序列表（与渲染同源）。
3. 补充回归测试并执行测试集验证。
4. 如出现问题，回滚到变更前 copy 拼装逻辑（单点回滚）。

## Open Questions

- 当前 `E` 导出是否应与 `C` 保持完全一致顺序；本次优先修复 `C`，若 `E` 共享同一构建路径则自然一致。
